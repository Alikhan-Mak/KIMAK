Project Context

This is an MVP prototype for a Smart City Management Dashboard focused on Almaty, Kazakhstan. Before implementing any feature, re-read the hackathon requirements document to ensure full compliance. Prioritize completeness over speed.

Architecture

Main Page — Interactive Map

  • ▸Display a detailed map of Almaty using a real tile-based provider (OpenStreetMap via Leaflet, or Mapbox) showing roads, buildings, and streets.
  • ▸Include ALL administrative districts of Almaty as overlay polygons.
  • ▸Support switchable data layers (one per domain).
  • ▸Use a continuous green → yellow → red gradient to represent load or severity. Thresholds are metric-specific, not always 0–100%.
  • ▸Clicking a district opens a detail panel with metrics and AI analysis.

Navigation

Top navigation bar with links to five domain dashboards. The currently active page must be visually highlighted.

AI Functionality

Incident Priority Model

Rank every incident using the following criteria, ordered from most to least important:

Level Criterion Examples
P1 🔴 Potential human casualties Gas leak, building collapse, fire
P2 🟠 Significant financial loss Substation failure, pipeline rupture
P3 🟡 Mass disruption of essential services District-wide water or heat outage
P4 🟢 Environmental damage Chemical spill, air quality breach
P5 🔵 Transport disruption Road closure, transit system failure
P6 ⚪ Public discomfort Low water pressure, street lighting fault

AI Response Requirements

For every selected district or incident the AI must answer three questions:

1

What is happening?

A factual summary of the current situation.

2

How critical is it?

The priority level (P1–P6) with a clear justification.

3

What actions should be taken?

Specific, actionable steps with responsible parties and estimated response times. Vague advice is not acceptable.

"Dispatch an emergency crew to substation #4 on Abay Ave within 30 minutes. Notify the 2,300 affected residents via SMS. Switch load to backup line B-12."

Conversational Follow-up

After the initial AI summary, display a chat input field so the user can ask follow-up questions about the same incident or district. Full conversation context must be preserved across turns.

Domain 5 — Tourism Monitor

Tracks visitor density, forecasts crowd load, and generates economic reallocation recommendations.

Metrics to Display

Metric Description
Visitor density by districtCurrent tourist load per district as a % of comfortable capacity
Top attraction loadReal-time occupancy for Shymbulak, Kok-Tobe, Panfilov Park, Medeu, Green Bazaar, etc.
3-day crowd forecastPredicted visitor volume per attraction for the next 72 hours
Redirection recommendationsAI-generated suggestions to move visitors from overloaded to underloaded sites
Economic impact estimateProjected revenue uplift for undervisited locations if redirection is applied
Sentiment scoreAggregated tourist satisfaction from mock review data
Average visit durationTime spent per attraction — detects bottlenecks

Redirection Recommendation Logic

When any attraction exceeds 80% capacity, the AI must:

  1. Identify alternative attractions with load below 50% and a similar visitor profile (nature, culture, food, etc.)
  2. Display a recommendation card:
Shymbulak 95% → redirect to Kok-Tobe 45%
Estimated revenue uplift for Kok-Tobe: +12%

3. Suggest the optimal time window for the redirect (e.g., "afternoon visitors most likely to reroute").

AI Recommendation Examples

"Deploy 2 additional shuttle buses from Almaty Arena to Medeu between 14:00–18:00 to absorb overflow from Shymbulak. Expected load reduction: 15%."
"Promote the Botanical Garden via the city app push notification to users currently near Kok-Tobe. Current Botanical Garden load: 28%."
"Flag Panfilov Park for crowd safety review — density has exceeded 90% for 3 consecutive hours."

Non-Functional Requirements

⚡

5-Second Rule

A city manager must understand overall city status within 5 seconds of opening the dashboard.

⏳

AI Loading State

All AI calls must display a loading state while processing.

🛡

Error Handling

All errors and empty states must be handled gracefully with informative fallback UI.

📱

Responsive Layout

Mobile-responsive is a bonus but not required for MVP.

📡

Telegram — Real-Time

All Telegram integrations must operate fully online and in real time.
